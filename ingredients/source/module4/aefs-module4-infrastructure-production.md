## Lessons

### 01 · Managed LLM Platforms — **Learn** · Python · ~60min
Maps the three hyperscaler bets: AWS Bedrock (marketplace — Claude/Llama/Titan behind one IAM + CloudWatch), Azure OpenAI (exclusive OpenAI partnership + Provisioned Throughput Units for dedicated capacity, ~50ms median vs Bedrock ~75ms on 405B-equivalent), and Vertex AI (Gemini-first, 1M-token multimodal differentiator). The real differentiator is the FinOps attribution surface — Bedrock Application Inference Profiles (cleanest native), Vertex project-per-team + BigQuery (most flexible), Azure scopes + tags (most opaque). Pushes a two-provider-minimum policy because single-hyperscaler lock-in locks you out of two-thirds of a frontier that now moves monthly.
**Tools/Infra:** AWS Bedrock, Azure OpenAI (PTUs), Vertex AI, CloudWatch / BigQuery Billing Export / Application Insights, VPC endpoints, BAAs.

### 02 · Inference Platform Economics — **Learn** · Python · ~60min
Bifurcates the 2026 inference market into custom silicon (Groq/Cerebras/SambaNova, 5–10x faster decode), GPU platforms (Baseten/Together/Fireworks/Modal/Anyscale), and API-first marketplaces (Replicate/DeepInfra). Positioning rule: Fireworks optimizes latency (FireAttention, raised $1/hr per GPU May 2026), Together optimizes catalog breadth, Baseten optimizes enterprise polish (Truss, $5B val), Modal optimizes Python-native DX (per-second billing), Replicate optimizes multimodal reach, Anyscale optimizes distributed Ray. Teaches a per-call economics comparator so $/M-token, $/minute, $/second, and $/prediction rate cards line up against a modeled workload.
**Tools/Infra:** Groq LPU, Cerebras WSE, SambaNova RDU, Baseten (Truss), Together, Fireworks (FireAttention), Modal, Replicate, Anyscale / RayTurbo.

### 03 · GPU Autoscaling on Kubernetes — **Learn** · Python · ~75min
Three layers, not one. Karpenter provisions nodes in ~45–60s (must switch consolidation from `WhenEmptyOrUnderutilized` to `WhenEmpty` or it evicts running GPU jobs mid-inference). KAI Scheduler handles gang scheduling + topology awareness + hierarchical queues (kills the 7-of-8 partial-allocation trap where seven nodes wait on one missing GPU). Application-level autoscalers (NVIDIA Dynamo Planner, llm-d WVA) scale on inference signals — queue depth, KV-cache utilization, goodput, P99 TTFT — never `DCGM_FI_DEV_GPU_UTIL` (a duty-cycle metric that reads 100% at 10 or 100 requests) and never memory (vLLM pre-allocates KV cache so HBM stays ~90% at one request).
**Tools/Infra:** Kubernetes, Karpenter, KAI Scheduler, kube-scheduler, NVIDIA Dynamo Planner, llm-d Workload Variant Autoscaler, HPA, DCGM metrics, Bottlerocket.

### 04 · vLLM Serving Internals — **Learn** · Python · ~75min
vLLM's dominance rests on three compounding defaults, not one trick: PagedAttention (block-table KV allocator, <4% fragmentation vs 60–80% classic, always-on), continuous batching (sequences join/leave the batch between decode iterations via the V1 scheduler), and chunked prefill (slices long prompts into ~512-token chunks so decode tokens never starve — drops P99 ITL from ~50ms to ~15ms). All three on pushes Llama 3.3 70B FP8 on one H100 to 2,200–2,400 tok/s at 128 concurrent (~3–4x a naive PyTorch loop). Ends with a toy continuous-batcher in `code/main.py` and the v0.18.0 gotcha: `--enable-chunked-prefill` won't compile with draft-model speculative decoding.
**Tools/Infra:** vLLM (V1 scheduler), PagedAttention, continuous batching, chunked prefill, KV block tables, `--gpu-memory-utilization`.

### 05 · EAGLE-3 Speculative Decoding in Production — **Learn** · Python · ~60min `[THREAD: eval]`
Speculative decoding pairs a fast draft head with the target; EAGLE-3 trains the draft on the target's hidden states (not raw tokens), pushing acceptance rate alpha to 0.6–0.8 on general chat. The only metric that matters is alpha on *your* traffic — below ~0.55 at high concurrency spec decode is net negative, because every rejected draft forces a second target forward. The production recipe is measure-first: ship plain, benchmark, enable EAGLE-3 via vLLM `speculative_config`, log `accepted_tokens_per_request`, and gate on P99 ITL (not P50 — mean drops while P99 can worsen on rejected-draft two-pass sequences). Domain-specific traffic drops alpha to 0.4–0.6 unless you train a domain draft head.
**Tools/Infra:** vLLM `speculative_config`, EAGLE-3 / EAGLE-2 / classic draft models, N-gram GPU spec decode, `spec_decode_metrics`, domain-trained draft heads.

### 06 · SGLang and RadixAttention for Prefix-Heavy Workloads — **Learn** · Python · ~75min
SGLang treats the KV cache as a reusable radix tree: a new request walks the tree, reuses KV blocks for any matching prefix, and prefills only the new suffix (~40x prefill savings on a shared 2K system prompt). Cache-aware depth-first scheduling + branch-level LRU keeps hot prefixes in HBM (FCFS evicts them). Edge: ~16,200 tok/s vs vLLM ~12,500 on ShareGPT (29%), up to 6.4x on prefix-heavy RAG, 86% cache hit on voice workloads; deployed on 400,000+ GPUs (xAI, LinkedIn, Cursor). The gotcha: the 6.4x evaporates unless prompt-template ordering is fixed — your prompt template *is* a cache key (system/tools/schemas first, question last; one deployment went 7%→74% hit rate by moving dynamic text out of the prefix).
**Tools/Infra:** SGLang, RadixAttention (radix-tree KV index), cache-aware scheduler, prefix-cache hit rate, LRU branch eviction.

### 07 · TensorRT-LLM on Blackwell with FP8 and NVFP4 — **Learn** · Python · ~75min
TRT-LLM is NVIDIA-locked but wins on Blackwell: SemiAnalysis InferenceX measured ~$0.012/M tokens on a 120B MoE via GB200 NVL72 + Dynamo vs ~$0.09/M on H100 + vLLM — a 7x economic gap. The stack compounds three float regimes: NVFP4 (4-bit microscaling) for weights/activations, FP8 still mandatory for KV cache (FP4 collapses attention scores), FP32 accumulator for softmax stability. Blackwell primitives: day-0 FP4 weights (no AWQ/GPTQ conversion step), integrated multi-token prediction, disaggregated prefill/decode, NVLink-5 all-to-all. Tradeoff is portability for throughput — and FP4 visibly degrades reasoning-heavy tasks, so validate per-model on your eval set before committing.
**Tools/Infra:** TensorRT-LLM, Blackwell B200 / GB200 NVL72, NVFP4 / MXFP4 microscaling, FP8 KV cache, NVIDIA Dynamo, NVLink 5, MLPerf Inference v6.0.

### 08 · Inference Metrics — TTFT, TPOT, ITL, Goodput, P99 — **Learn** · Python · ~60min `[THREAD: eval]`
The metrics that decide whether a deployment works: TTFT (prefill + queue + network), TPOT/ITL (memory-bound decode per token), E2E (TTFT + TPOT × output length), throughput (fleet tokens/s). The one that matters for product is **goodput** — the fraction of requests meeting every SLO constraint simultaneously; high throughput at 60% goodput is failure. Always report P50/P90/P99, never mean (LLM latency is right-skewed and users hit P99 regularly). Watch the measurement trap: GenAI-Perf excludes TTFT from ITL, LLMPerf includes it — two tools disagree on TPOT for the same run. Reference (Llama-3.1-8B on TRT-LLM): mean TTFT 162ms / TPOT 7.33ms / E2E 1,093ms.
**Tools/Infra:** TTFT / TPOT / ITL / E2E / goodput, GenAI-Perf, LLMPerf, P50/P90/P99, multi-constraint SLOs, MLPerf Inference v6.0.

### 09 · Production Quantization — **Learn** · Python · ~75min
Quantization format is a function of hardware + engine + workload, not a universal choice. The 2026 zoo: GGUF Q4_K_M/Q5_K_M (CPU/edge via llama.cpp/Ollama), GPTQ (multi-LoRA on vLLM, Marlin kernels), AWQ (datacenter GPU default — Marlin-AWQ ~741 tok/s on 7B, best INT4 Pass@1), FP8 (near-lossless middle on Hopper/Ada/Blackwell), MXFP4/NVFP4 (Blackwell aggressive, no LoRA support yet). Two traps: the calibration dataset must match deployment domain (generic C4 lets AWQ protect the wrong weights and HumanEval drops points), and weight quant ≠ KV quant — "my model is 4GB now" forgets the 10–30GB KV cache at production batch.
**Tools/Infra:** AWQ + Marlin-AWQ, GPTQ + Marlin, GGUF K-quants (llama.cpp/Ollama), FP8, MXFP4/NVFP4, calibration datasets (C4/WikiText).

### 10 · Cold Start Mitigation for Serverless LLMs — **Learn** · Python · ~60min
A 20GB image takes 3–8 minutes (70B) to go from cold to serving — an outage, not a warm-up. Six mitigation layers: pre-seeded node images (Bottlerocket dual-volume snapshots, −2–4min), model streaming (NVIDIA Run:ai Model Streamer, native in vLLM 2026, overlaps I/O with compute), GPU memory snapshots (Modal checkpoints, up to 10x restart), warm pools (`min_workers=1`), tiered loading (ServerlessLLM NVMe→DRAM→HBM, 10–200x), and live migration that ships input tokens (KB) not KV cache (GB). The warm-pool arithmetic: brutal on small models ($0.85–1.50/hr to skip a 30s cold start), kind on large ones; mandatory once P99 TTFT SLA < 60s on 70B+.
**Tools/Infra:** Bottlerocket, NVIDIA Run:ai Model Streamer, Modal GPU checkpoints, ServerlessLLM, Karpenter, warm pools (`min_workers`), live migration.

### 11 · Multi-Region LLM Serving and KV Cache Locality — **Learn** · Python · ~60min
Round-robin load balancing is actively harmful for cached inference — a request that misses its prefix pays full prefill (~800ms vs ~80ms on a cache hit, a 10x TTFT gap). The production pattern is a cache-aware router (vLLM Router in Rust, llm-d router) that subscribes to KV-cache events and routes on prefix-hash match, falling through to least-queue-depth. Cross-region adds a network term (us-east↔ap-southeast ~220ms RTT) — GORGO jointly minimizes prefill + network latency, so stay regional unless prefill dominates. DR reality: 32% of LLM DR failures come from backing up weights but forgetting `tokenizer.json`, quant configs, or engine configs.
**Tools/Infra:** vLLM Router (Rust), llm-d router, prefix-hash routing, KV-cache event pub/sub, GORGO, Bedrock cross-region inference, GKE Multi-Cluster Gateway.

### 12 · Edge Inference — **Learn** · Python · ~60min
The edge ceiling is memory bandwidth, not compute — mobile DRAM at 50–90 GB/s vs datacenter HBM3 at 2–3 TB/s (30–50x gap), and decode is bandwidth-bound so the gap is decisive. Four targets: Apple Neural Engine (M4/A18, 38 TOPS, unified memory, Core ML INT4 + FP16), Qualcomm Hexagon (Snapdragon X Elite/8 Gen 4, 45 TOPS, QNN INT8/INT4), WebGPU + WebLLM (Llama 3.1 8B Q4 ~41 tok/s on M3 Max, ~70–75% mobile coverage, Safari iOS 26 GA), NVIDIA Jetson (AGX Orin runs gpt-oss-20b via vLLM at ~40 tok/s; Thor/T4000 doubles it). TensorRT Edge-LLM ports EAGLE-3, NVFP4, and chunked prefill to the edge.
**Tools/Infra:** Apple Neural Engine + Core ML / MPS, Qualcomm QNN / AI Hub, WebGPU / WebLLM, NVIDIA Jetson Orin / Thor, TensorRT Edge-LLM, OpenVINO.

### 13 · LLM Observability Stack Selection — **Learn** · Python · ~60min `[THREAD: eval]`
The 2026 observability market splits into development platforms (LangSmith, Langfuse, Comet Opik — bundle monitoring with evals, prompt management, session replay) and gateway/telemetry tools (Helicone, SigNoz, OpenLLMetry, Phoenix — traces/metrics only, OTel-glueable). Langfuse (MIT core, 50K events/mo free) is the OSS-balance pick; Phoenix (Elastic 2.0) is OTel-native and best for RAG/drift viz but not a persistent backend; Arize AX uses zero-copy Iceberg/Parquet claiming ~100x cheaper than monolithic observability at scale; LangSmith leads only if committed to LangChain/LangGraph ($39/user/mo). Common production pattern: a gateway (Helicone/Portkey) + a separate eval platform (Phoenix/TruLens) glued by OpenTelemetry.
**Tools/Infra:** Langfuse, LangSmith, Phoenix (Arize), Arize AX, Helicone, Comet Opik, OpenLLMetry, SigNoz, OpenTelemetry (GenAI semconv), TruLens.

### 14 · Prompt Caching and Semantic Caching Economics — **Learn** · Python · ~60min
Two caching layers. L2 provider prompt/prefix caching reuses attention KV — Anthropic `cache_control` gives 10x cheaper reads ($0.30/M vs $3.00/M fresh, 5-min or 1-hour TTL with write premiums), OpenAI auto-caches prompts ≥1024 tokens at ~90% discount. L1 app-level semantic caching (Redis Vector/GPTCache/Qdrant, cosine ≥0.95) bypasses the LLM entirely — production hit rates range 10% (open chat) to 70% (structured FAQ); vendor "95% accuracy" means match-correctness, not hit rate. Two pitfalls that collapse savings: parallelization (N parallel requests before the first cache write = N writes, 0 reads, 5–10x bill) and dynamic content inside the prefix (timestamps/IDs → 0% hits).
**Tools/Infra:** Anthropic `cache_control`, OpenAI prompt caching, Gemini context caching, GPTCache, Redis Vector Similarity, Qdrant, Portkey/Helicone Cache, RadixAttention (self-hosted).

### 15 · Batch APIs — the 50% Discount as Industry Standard — **Learn** · Python · ~45min
Every major provider ships an async batch API with a 50% discount and ~24-hour turnaround (usually 2–6h in practice). Stack batch + prompt caching on an overnight pipeline and cost drops to ~10% of synchronous-uncached ($2,000/night → $180/night on a 50k-doc summarization). The rule: if it's not interactive, it belongs on batch. Triage every workload into three lanes — interactive (sync + caching), semi-interactive (async queue + fallback), batch (overnight + cached input). Two traps: partial interactivity (features that tolerate 10min but ship sync, wasting 10x) and output-schema drift (OpenAI/Anthropic/Google batch file formats differ per provider).
**Tools/Infra:** OpenAI Batch API, Anthropic Message Batches, Vertex AI Batch Prediction, Fireworks/Together batch tiers, JSONL batch files, `cache_control` in batch.

### 16 · Model Routing as a Cost-Reduction Primitive — **Learn** · Python · ~60min `[THREAD: versioning]` `[THREAD: eval]`
A dynamic broker evaluates each request and sends simple queries to a cheap model, escalating complex ones to a frontier model (model cascading) — production case studies show 20–60% cost reduction at iso-quality. Four routing signals: task classification, prompt length, embedding similarity to a known-hard set, and self-confidence from the first pass. Three patterns: pre-route (classifier up front, fastest), cascade (cheap-first + confidence check, best quality floor), ensemble (parallel, A/B only). The real risk is cheap-model drift — the route pushes 40% to a weaker model, quality drops 3–5%, nobody notices for a quarter — so gate routes by online quality metrics (thumbs-up, LLM-judge on a 5% holdout, escalation rate, refusal rate), not just offline eval sets.
**Tools/Infra:** LiteLLM router, Portkey, Kong AI Gateway, OpenRouter, RouteLLM (LMSYS), Not Diamond, embedding-similarity classifiers, online quality gates.

### 17 · Disaggregated Prefill/Decode — **Learn** · Python · ~75min
Prefill is compute-bound; decode is memory-bound; colocating them on one GPU wastes 20–40% of GPU time buying the wrong resource. Disaggregation splits them onto separate pools and transfers KV cache over NIXL (RDMA/InfiniBand, TCP fallback) — transfer tax is ~20–80ms for a 4K-token 70B FP8 prompt, so short prompts don't justify it. NVIDIA Dynamo (1.0 GA) sits above vLLM/SGLang/TRT-LLM with a Planner Profiler + SLA Planner that auto-rate-matches prefill:decode (NVIDIA reports ~6x DeepSeek-R1 MoE on GB200; community "30x" lacks a single primary source). llm-d (Red Hat + AWS) is the Kubernetes-native equivalent with per-role HPA and `packDomain: rack` topology.
**Tools/Infra:** NVIDIA Dynamo (Planner/SLA Planner), llm-d, NIXL (RDMA/InfiniBand), GB200 / GB300 NVL72, UCCL, prefill/decode pool split.

### 18 · vLLM Production Stack with LMCache KV Offloading — **Learn** · Python · ~60min
The reference Kubernetes deployment: router (cache-aware) + engines (vLLM workers) + KV offload + observability + control plane, shipped as Helm chart + operator. LMCache extracts KV cache to CPU DRAM (then Ceph/disk) so preempted requests restore from CPU instead of re-prefilling, and repeated prefixes share across engines — the vLLM 0.11.0 KV Offloading Connector (Jan 2026) makes this asynchronous and pluggable via the Connector API (v0.9.0+). Benchmark behavior on 16x H100: at low KV footprint LMCache adds ~3–5% overhead (skip it), but once KV exceeds HBM, native CPU offload and LMCache both improve throughput substantially (LMCache more, via cross-engine sharing). Decisive for multi-tenant / RAG with shared system prompts.
**Tools/Infra:** vLLM production-stack (router / engines / operator), LMCache, KV Offloading Connector API (v0.9.0+ / 0.11.0), Ceph/S3, Prometheus / Grafana / OTel, Helm.

### 19 · AI Gateways — **Learn** · Python · ~60min `[THREAD: versioning]`
A gateway sits between apps and providers, consolidating seven features into one OpenAI-compatible API: provider routing, fallback (OpenAI 429 → Anthropic), retries, rate limits, secret references (vault at runtime), observability + cost attribution, and guardrails. 2026 split: LiteLLM (MIT OSS, 100+ providers, breaks down ~2000 RPS / 8GB), Portkey (Apache 2.0 since Mar 2026, guardrails/PII/audit, 20–40ms overhead, $49/mo), Kong AI Gateway (Kong's own benchmark: 228% faster than Portkey, 859% faster than LiteLLM on 12 CPUs; $100/model/mo), Bifrost (Maxim AI, configurable retry/fallback). Data residency is the forcing function that sends healthcare/finance to self-host.
**Tools/Infra:** LiteLLM, Portkey, Kong AI Gateway, Bifrost, Cloudflare/Vercel AI Gateway, HashiCorp Vault, OpenTelemetry, guardrails (PII/jailbreak).

### 20 · Shadow Traffic, Canary Rollout, and Progressive Deployment — **Learn** · Python · ~60min `[THREAD: versioning]` `[THREAD: eval]`
LLM rollouts combine the hardest parts of deployment: no unit tests, diffuse failure modes, delayed signals. The sequence is shadow mode (duplicate prod traffic to candidate, log + diff, zero user impact — catches cost blow-ups and length regressions but NOT quality), canary (1%→10%→25%→50%→75%→100% gated on five metrics: latency percentiles, cost/request, error/refusal rate, output-length distribution, user feedback), then A/B for distinct alternatives. Non-determinism is irreducible (up to 15% accuracy variation across identical runs from GPU FP non-associativity + batch-size variance), so "stable" means within-variance, not identical. Rollback is the weapon: policy lives in flags, model lives in registry with pinned digests — flip + revert + pin in seconds, not hours.
**Tools/Infra:** Argo Rollouts, Flagger, Istio/Linkerd weighted routing, feature flags, model registry (pinned digests), shadow traffic mirroring.

### 21 · A/B Testing LLM Features — **Learn** · Python · ~60min `[THREAD: eval]`
Traditional A/B testing wasn't built for non-deterministic LLMs. The critical split: evals answer "can the model do the job?" (offline, labeled, judge), A/B tests answer "do users care?" (online, randomized, user-level metric) — both required, shipping on vibes is over. Three testable axes: prompt engineering, model selection (accuracy vs cost vs latency, multi-objective), generation parameters. Methodology: CUPED (30–70% variance reduction via pre-period regression), sequential testing (always-valid mSPRT / Howard confidence sequences for peek-and-decide), Benjamini-Hochberg / Bonferroni multiple-comparison corrections, SRM checks. Platform split: Statsig (acquired by OpenAI $1.1B Sept 2025, all-in-one SaaS) vs GrowthBook (OSS MIT, warehouse-native, Bayesian + Frequentist + Sequential engines).
**Tools/Infra:** Statsig, GrowthBook, CUPED, sequential testing (mSPRT), Benjamini-Hochberg / Bonferroni, SRM checks, feature flags.

### 22 · Load Testing LLM APIs — Why k6 and Locust Lie — **Build** · Python · ~75min `[THREAD: eval]`
Generic load testers lie for LLM APIs via two traps. The GIL trap: Locust runs tokenization client-side under the Python GIL, so under concurrency the tokenizer queues behind request generation and inflates reported ITL — your client is the bottleneck, not the server. The prompt-uniformity trap: identical prompts in a loop hit ~100% prefix cache and test one point on the token distribution; LLMPerf fixes it with `--mean-input-tokens` + `--stddev-input-tokens`. Tool mapping: LLMPerf (Rust-backed tokenization, benchmark default), NVIDIA GenAI-Perf (reference, but excludes TTFT from ITL — disagrees with LLMPerf on TPOT), guidellm (large-scale synthetic), and k6 v2026.1.0 + k6 Operator 1.0 GA (streaming-aware, Kubernetes-native via TestRun/PrivateLoadZone CRDs, best for CI/CD gates). Four patterns: steady, ramp, spike (autoscaling), soak (memory leaks).
**Tools/Infra:** k6 v2026.1.0 + k6 Operator 1.0 GA, LLMPerf, NVIDIA GenAI-Perf, LLM-Locust, guidellm, Vegeta, Locust 2.43.3.

### 23 · SRE for AI — **Learn** · Python · ~60min `[THREAD: eval]`
AI SRE grounds LLMs in infrastructure data (logs, runbooks, service topology) via RAG to automate the first ~20min of incident investigation. The 2026 pattern is multi-agent orchestration: a supervisor dispatches to specialized agents (logs, metrics, runbooks), synthesizes a hypothesis + evidence, and a human approves before any action — auto-remediation stays narrow (restart pod, revert specific deploy, scale within policy) and anyone selling "set it and forget it" is overselling. NeuBird Hawkeye uses adversarial evaluation (two models analyze the same incident; agreement = confidence, disagreement = escalate). MIT 2025: an LLM on historical logs + GPU temps + API error patterns predicted 89% of outages 10–15min early — but predictions without actuation are just dashboards.
**Tools/Infra:** Datadog Bits AI, Azure SRE Agent, NeuBird Hawkeye, multi-agent supervisor pattern, RAG over runbooks/postmortems, Prometheus / Loki.

### 24 · Chaos Engineering for LLM Production — **Learn** · Python · ~60min `[THREAD: eval]`
Chaos for LLM stacks is its own discipline — new failure modes don't show up in unit tests. Five prerequisites before running in prod: defined SLI/SLO, trace+metric+log observability, automated rollback, runbooks, on-call (missing any turns chaos into a real incident). Four planes (control / target / safety / observability) + feedback into SLO; guardrails are mandatory — burn-rate alerts pause experiments if daily error-budget burn > 2x expected, suppression windows + trace-ID correlation dedupe alert noise. Five LLM-specific experiments: memory/KV-preemption overload, network failure, provider outage simulation (100% 429), malformed/tokenizer-stalling prompts, KV eviction storms. Cadence: weekly canary, monthly game day + postmortem, quarterly cross-team audit.
**Tools/Infra:** Harness Chaos Engineering, LitmusChaos (CNCF), Chaos Mesh (CNCF), burn-rate alerts, error-budget gates, trace-ID correlation, SLI/SLO.

### 25 · Security — Secrets, API Key Rotation, Audit Logs, Guardrails — **Learn** · Python · ~60min
Centralize secrets (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault) — never config files in VCS, env files, spreadsheets, or static keys; the 2026 pattern is apps → gateway → provider with the gateway pulling credentials from vault at runtime, so rotation propagates in minutes with no redeploys (≤90-day rotation policy; TruffleHog / GitGuardian / Gitleaks on every commit). PII/PHI scrubbing uses entity recognition (spaCy NER, Presidio) + consistent tokenization (Mesh approach — same value → same placeholder) so the LLM preserves code/relationship semantics. Network egress: LLM pods in a dedicated subnet whitelisting only `api.openai.com` / `api.anthropic.com`. The 2026 incident driver: the Vercel supply-chain attack exfiltrated env vars across thousands of deployments via compromised CI/CD credentials.
**Tools/Infra:** HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, TruffleHog / GitGuardian / Gitleaks, spaCy NER / Presidio, zero-trust (MFA/SSO/RBAC/ABAC), VPC egress whitelisting.

### 26 · Compliance — SOC 2, HIPAA, GDPR, PCI-DSS, EU AI Act, ISO 42001 — **Learn** · policy/process · ~60min
Multi-framework coverage is table stakes for 2026 enterprise deals. EU AI Act: in force Aug 2024, high-risk enforcement Aug 2 2026, fines up to €35M/7% (prohibited practices) or €15M/3% (high-risk obligations), applies globally if serving EU users; Colorado AI Act effective June 30 2026. SOC 2 Type II (not Type I) is the fintech gate. GDPR: real-time inference-layer PII redaction is the defensible standard — post-processing cleanup means the model already saw the data (largest AI-specific fine: €30.5M Clearview AI; OpenAI's €15M Garante fine overturned on appeal Mar 2026). HIPAA requires a signed BAA before any PHI leaves infra. ISO 42001 (AI governance) is the emerging procurement requirement alongside ISO 27001. Cross-framework control mapping (access control → ISO 27001 A.5.15-5.18 + GDPR Art. 32 + HIPAA §164.312(a)) reduces audit fatigue.
**Tools/Infra:** SOC 2 Type II, HIPAA BAAs, GDPR Art. 30/32, PCI-DSS, EU AI Act (Art. 99 fines), Colorado AI Act, ISO 42001 / 27001, real-time PII redaction.

### 27 · FinOps for LLMs — **Learn** · Python · ~60min
Traditional FinOps (tags + tiers) breaks on LLM spend — costs are token-transactions not resource-uptime, tags don't auto-propagate to line items, and engineering decisions (prompt design, context window, output length) are financial decisions. Three attribution dimensions to instrument at call-site on day one: per-user (`user_id`), per-task (`task_id` + route), per-tenant (`tenant_id`) — retroactive tagging always misses. Four token layers (prompt 40–60%, tool 20–40%, memory 10–30%, response 10–30%) — one bucket hides spend. Enforcement ladder for multi-tenant: rate limit (2–3x peak, 429 + Retry-After) → daily spend cap (1.5–3x ceiling) → kill switch on spend z-score > 4 (auto-pause + page on-call). Unit metric is cost per resolved query / per artifact, not $/M tokens.
**Tools/Infra:** telemetry-joiner (trace-ID → billing), tag-and-aggregate, sampling-and-extrapolation, event-sourced (Kafka/Kinesis), rate limits / spend caps / kill switches, per-user/task/tenant attribution.

### 28 · Self-Hosted Serving Selection — **Learn** · Python · ~45min
Four engines dominate self-hosted inference in 2026; pick by hardware → scale → workload. llama.cpp (fastest on CPU, widest model support), Ollama (dev-laptop one-command, 15–30% slower than llama.cpp, 3x prod throughput gap), TGI (entered maintenance mode Dec 11 2025 — risky long-term bet, biases new projects to vLLM/SGLang), vLLM (general-purpose production default, v0.15.1 Feb 2026), SGLang (agentic multi-turn / prefix-heavy specialist, 400,000+ GPUs). Hardware forcing functions: CPU-only → llama.cpp; AMD / non-NVIDIA → vLLM only (TRT-LLM is NVIDIA-locked); Blackwell → TRT-LLM throughput leader. Pipeline pattern: dev = Ollama, staging = llama.cpp, prod = vLLM or SGLang, same GGUF / HF weights throughout.
**Tools/Infra:** llama.cpp, Ollama, TGI (maintenance mode), vLLM, SGLang, TensorRT-LLM, GGUF / HF weights, AMD ROCm.
