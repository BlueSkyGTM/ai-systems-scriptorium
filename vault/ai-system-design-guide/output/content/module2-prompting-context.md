### 01 · Prompt Engineering Fundamentals
This section establishes prompt engineering as a disciplined architectural practice rather than trial-and-error writing, centered on maximizing intent disclosure while minimizing output variance. It details the tiered message instruction hierarchy (System, Developer, User, Assistant) used in production systems to manage safety, technical overrides, and conversational state. The section covers role prompting as a capabilities anchor, structural delimiter usage for context separation, and strategic selection between zero-shot and few-shot approaches depending on model scale and reasoning capabilities.
**Key concepts:** Intent Disclosure, Output Variance, Instruction Hierarchy, Role Prompting, Capabilities Anchor, Delimiters, Zero-Shot prompting, Few-Shot prompting, Chain-of-Thought (CoT), Programmatic CoT

### 02 · Few-Shot and In-Context Learning (ICL)
In-Context Learning (ICL) enables an LLM to perform new tasks from examples provided in the prompt without weight updates, making example construction and selection a primary lever for prompt stability. A well-formed few-shot example consists of realistic input, optional reasoning, and a gold-standard output, with optimal example counts scaling inversely with model size (5–10 for small models down to 1–2 for frontier models). Dynamic example selection via semantic search over a vector DB of gold examples replaces static prompt templates in production systems, injecting only locally relevant cases per query. Critical tradeoffs include context-window latency, attention dilution ("lost-in-the-middle"), and label distribution bias, which must be mitigated through label balancing, example shuffling, and permutation testing.

**Key concepts:** In-Context Learning (ICL), few-shot prompting, example anatomy (input/reasoning/output), model-size-dependent example scaling, dynamic example selection, semantic similarity retrieval, vector DB of gold examples, label balancing, distribution bias, analogy prompting, few-shot chain-of-thought, attention dilution, permutation testing

### 03 · Chain-of-Thought (CoT)
Chain-of-Thought (CoT) elicits intermediate reasoning steps from an LLM before producing a final answer, providing working memory for complex logic and math tasks. The technique spans Zero-Shot CoT (triggered by "let's think step by step"), Few-Shot CoT (example-based), and Programmatic CoT (explicit multi-step instructions for agents), with modern reasoning models like o1 and DeepSeek-R2 now baking CoT into the architecture via reinforcement learning using dedicated hidden thinking windows. Production systems increasingly layer self-verification (generate, critique, correct) and execution-verified CoT (write, run, fix code) on top of single chains, while hybrid reasoning architectures mitigate the latency, cost, and over-thinking failure modes by classifying query complexity and routing to appropriate tiers. CoT follows the Inference Scaling Law where longer thinking yields better problem-solving on hard tasks, but adds token cost and can cause hallucinated complexity on simple queries.

**Key concepts:** Zero-Shot CoT, Few-Shot CoT, Programmatic CoT, Thinking Models (o1, DeepSeek-R2, Claude extended thinking), Hidden CoT, Inference Scaling Law, Self-Verification, Execution-Verified CoT, Over-thinking, Hybrid Reasoning Architecture, Knowledge Distillation, Working Memory / Scribble Pad

### 04 · Tree-of-Thought (ToT)
Tree-of-Thought (ToT) is a branching prompting architecture where a model explores multiple reasoning paths in parallel, evaluates intermediate steps, and backtracks from dead ends, contrasting with the linear, single-path approach of Chain-of-Thought. The architecture operates via a loop consisting of a thought proposer (generating 3-5 next steps), a state evaluator (grading steps as viable or impossible), and a search algorithm like BFS or DFS to traverse the branches. ToT is specifically designed to prevent hallucination cascades by catching errors early via the evaluator, and it has evolved into Monte Carlo Tree Search (MCTS) implementations that scale search-time compute using specialized "Searcher" models. The primary tradeoff is exponential cost and latency—exploring branches to depth can require 15-20 LLM calls for a single query—making it better suited for high-stakes offline tasks like software refactoring, puzzle solving, or generating golden datasets rather than real-time consumer applications.
**Key concepts:** Tree-of-Thought (ToT), branching topology vs. linear topology, Thought Proposer, State Evaluator, Search Algorithm (BFS/DFS), hallucination cascades, backtracking and self-correction, Monte Carlo Tree Search (MCTS), search-time compute scaling, RAD-T (Reasoning-as-Data-Tree), Searcher models.

### 05 · Context Engineering
Context engineering addresses the continuous curation of tokens within an LLM's finite context window, shifting the focus from merely fitting data to ranking relevance and managing the compute budget across multi-turn agent loops. The section details five core techniques—compaction, just-in-time loading, structured note-taking, sub-agent isolation, and system prompt calibration—used to mitigate "context rot" (accuracy degradation as token counts grow) and keep only the highest-signal tokens in the window. It covers long context paradigms (In-Context RAG), Extended Thinking parameters (budget tokens and reasoning effort), persistent challenges like the "Lost-in-the-Middle" attention gradient, and economic optimizations including context budgeting, prefix caching, and Reasoning-Aware Deletion (RAD-L) for massive prompts.
**Key concepts:** Context rot, Agentic context engineering, Compaction, Just-in-time loading, Structured note-taking, Sub-agent isolation, System prompt calibration, In-Context RAG, Extended Thinking, Budget tokens, Reasoning effort, Lost-in-the-Middle, Attention gradient, Context budgeting, Prefix caching, Contextual compression (RAD-L)

### 06 · Structured Generation
Structured Generation forces LLMs to produce machine-readable output (JSON, YAML, CSV) with 100% syntactic reliability by shifting from prompt-based requests to engine-level constraints such as JSON Mode, function calling, and constrained decoding (CFG/Regex). JSON Mode applies a logit bias or grammar mask at the inference level, restricting token selection to only schema-valid characters, while self-hosted models use libraries like Outlines with context-free grammars or regex patterns. For complex extraction tasks involving many hierarchical fields, a multi-stage pattern is recommended: a large model first extracts facts in natural language, then a smaller model converts them to strict JSON, mitigating "omission hallucinations" caused by the model dividing attention between structural compliance and data accuracy. Even with guaranteed syntactic validity, semantic validation via Pydantic/Zod remains necessary, using traceback-based retry loops where validation errors are fed back to the model for correction.
**Key concepts:** JSON Mode, response_format, function calling, parallel function calling, constrained decoding, Context-Free Grammars (CFG), regex-guided generation, Outlines, multi-stage extraction, Chain-of-Density extraction, Pydantic/Zod validation, logit bias, grammar masking, schema complexity vs. information integrity tradeoff, omission hallucinations.

### 07 · Prompt Optimization (DSPy) [THREAD: versioning]
DSPy abstracts prompt engineering into a programmatic compilation pipeline by separating declarative logic (Signatures and Modules like `ChainOfThought`) from prompt formatting and optimization. Instead of manually hand-tuning text instructions, developers define input/output schemas and a scoring metric, then rely on automated Teleprompters (optimizers) such as `BootstrapFewShot` and the Bayesian-based `MIPROv2` to algorithmically discover the most effective instructions and few-shot examples for a specific model. This approach treats prompts as learnable weights (the "Prompt as Weight" analogy), enabling systems to be deterministically re-compiled or re-optimized if the underlying model is swapped. The tradeoff is the requirement of defining robust evaluation metrics and providing training data to drive the automated optimization loops. **Key concepts:** DSPy, Signatures, Modules, Teleprompters, BootstrapFewShot, MIPROv2, Prompt as Weight analogy, Metric-Driven Optimization, LLM-as-Judge, compilation.

### 08 · Prompt Injection and Defense
Prompt injection is framed as the LLM-era equivalent of SQL injection (OWASP LLM Top 10 risk #1), encompassing both direct attacks and indirect attacks via external content read by agents. Defenses are treated as architectural rather than prompt-writing concerns, including the Dual-LLM security proxy pattern (small guard model filtering input before it reaches the frontier logic model), XML-based input isolation using untrusted data tags with H-Rank weighting, canary-token output filtering for system-prompt leakage, and format-hijacking prevention. For agentic systems, the section covers privilege-escalation risks mitigated via Human-in-the-Loop (HITL) controls on sensitive tools and least-privilege token scopes. RAG-specific indirect injection is addressed by treating retrieved chunks as untrusted data processed through a separate analyzer pass before final generation.
**Key concepts:** Direct Injection, Indirect Injection, Dual-LLM Defense Pattern, Security Proxy, Guard Model, Input Isolation, XML tags, H-Rank training, Canary Tokens, Format Hijacking, Autonomous Privilege Escalation, Human-in-the-Loop (HITL), Least Privilege, Semantic Defense, Untrusted Data handling in RAG
