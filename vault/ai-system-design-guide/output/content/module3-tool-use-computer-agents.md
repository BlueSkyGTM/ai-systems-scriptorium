### 01 · The 2026 Tool-Use and Computer Agent Landscape [THREAD: safety]
The 2026 tool-use ecosystem has consolidated around four primary architectural patterns: local agents, cloud agents, IDE agents, and vision-based computer-use agents. These categories are increasingly converging, blurring the lines between local execution and cloud-hosted models through standardized protocols like MCP. The landscape features prominent tools such as OpenClaw (a viral personal assistant), OpenHands (a sandboxed autonomous developer), Claude Computer Use (vision-based GUI automation), Claude Code (a terminal agent), and IDE-integrated agents like Cursor and Windsurf. Key industry trends include the non-negotiable adoption of sandboxing due to widespread security incidents, cost optimization via plan-and-execute models, and a shift toward asynchronous background agents. 
**Key concepts:** Local Agents, Cloud Agents, IDE Agents, Computer-Use Agents, MCP (Model Context Protocol), Sandboxed Execution, SOUL.md Identity Layer, Event-Stream Architecture, Vision-Action Loop, Background Agents

### 02 · Architecture Patterns for Tool-Use Agents [THREAD: safety]
Production tool-use agents are built on four core architecture patterns: function/tool calling (structured API execution), vision-based automation (screenshot observation and action loops), local code execution (natural language to executable code), and multi-agent orchestration (specialized agents routed by a planner). Critical system design decisions revolve around the execution boundary—favoring isolated, sandboxed environments (Docker, microVMs) over unsandboxed host access to mitigate catastrophic risks—and implementing robust state management and self-correcting error loops. Furthermore, agents increasingly standardize tool integration via the Model Context Protocol (MCP) using direct connections, multi-server fan-outs, or enterprise gateways, while optimizing costs through hierarchical model routing.
**Key concepts:** Function/Tool Calling, Vision-Based Automation (Observe-Reason-Act Cycle), Local Code Execution, Permission Gates, Multi-Agent Orchestration, Plan-and-Execute Strategy, Sandboxed vs. Unsandboxed Execution, State Management Patterns (Conversation, Session, Persistent, Environment), Self-Correction Loops, Error Taxonomy, MCP Integration Patterns (Direct, Multi-Server Fan-Out, MCP Gateway), Dynamic Tool Manifests.

### 03 · OpenClaw Deep Dive: The Open-Source Personal AI Agent [THREAD: safety]
OpenClaw is a self-hosted, open-source personal AI agent that interacts via 20+ messaging platforms using a single Gateway process to route normalized messages to an LLM-powered Agent Runtime. It extends LLM capabilities through an AgentSkills system, where modular directories containing YAML frontmatter define triggers, available tools, and execution behaviors. Security and control are managed via a defense-in-depth model featuring channel authentication, agent-level tool allowlists, sandbox policies, and Docker isolation with restricted networking to prevent data exfiltration. The architecture relies on a single-operator trust model that provides unified, cross-channel memory and state persistence without the complexity of distributed databases, but it lacks the rigorous isolation required for multi-tenant SaaS deployments.

**Key concepts:** self-hosted personal AI agent, messaging-native interface, Gateway WebSocket server, Channel Adapters, Agent Runtime, multi-agent routing, AgentSkills system, YAML frontmatter triggers, tool execution, selective skill injection, LLM provider abstraction, sandbox isolation, defense-in-depth permission layers, Docker network restrictions, unified cross-channel memory, file-based memory system, provider policy risk management, single-operator trust model

### 04 · Computer-Use Agents
Computer-use agents operate graphical interfaces by interpreting raw pixel screenshots and issuing low-level mouse and keyboard commands, rather than relying on structured APIs. They execute tasks through a continuous screenshot-reason-act loop, requiring isolated sandbox environments—such as Docker containers with virtual displays—to securely manage OS control and prevent unauthorized access. While they are slower and more expensive per action than traditional automation tools like Selenium, they provide superior resilience against UI changes and anti-bot detection by visually adapting to dynamic layouts. ### 07 · Safety and Governance for Tool-Using Agents [THREAD: safety]
Tool-using agents introduce qualitatively different risks than chatbots because language model errors are converted directly into real-world actions, such as data deletion or exfiltration. The chapter establishes a defense-in-depth architecture based on the OWASP Top 10 for Agentic Applications, emphasizing that models may bypass their own safety reasoning under pressure, making system-level controls mandatory. It details critical security mechanisms including input sanitization for indirect prompt injection, capability-based access control, hardware-level sandboxing (e.g., Firecracker, gVisor), and tiered human-in-the-loop (HITL) approval gates. Finally, it outlines comprehensive enterprise governance frameworks, audit logging, multi-level kill switches, and regulatory compliance (EU AI Act) necessary for safe production deployments.
**Key concepts:** Defense-in-depth architecture, OWASP Top 10 for Agentic AI, PropensityBench pressure testing, indirect prompt injection, tool output sanitization, data loss prevention (DLP), cascading failure containment, isolation technologies (Docker, gVisor, WASM, Firecracker), capability-based access control, allowlists vs. denylists, tiered authorization models, Human-in-the-Loop (HITL) approval gates, rate limiting, immutable audit logging, multi-level kill switches, governance maturity models, EU AI Act compliance.
