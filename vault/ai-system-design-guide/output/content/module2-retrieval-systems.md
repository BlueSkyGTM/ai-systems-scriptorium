### 01 · RAG Fundamentals  [foundational RAG]
Retrieval-Augmented Generation (RAG) is the architectural pattern of grounding LLM responses with external, verifiable context, replacing fine-tuning for factual updates with zero-cost database updates. The RAG taxonomy spans four levels of agentic depth—Naive (vector search, deprecated for production), Advanced (query transformation, hybrid search, RRF, reranking), Agentic (stochastic loops like Self-RAG/CRAG), and GraphRAG (knowledge graph traversal for aggregative questions). With modern long-context windows (2M+ tokens), architects choose between Standard RAG for dynamic corpora over 100k tokens, or In-Context RAG (ICR) with prompt caching for smaller static corpora under 50k tokens. Production failures are primarily caused by three retrieval gaps: semantic mismatch, missing context, and lost-in-the-middle, requiring targeted retrieval strategies rather than prompt tuning to resolve.
**Key tools/concepts:** Retrieval-Augmented Generation (RAG), RAG Taxonomy (Naive, Advanced, Agentic, GraphRAG), Fine-Tuning vs. Grounding, Reciprocal Rank Fusion (RRF), Self-RAG, Corrective RAG (CRAG), In-Context RAG (ICR), Prompt Caching, Semantic Mismatch Gap, Missing Context Gap, Lost-in-the-Middle Gap, Knowledge Graphs.

### 02 · Chunking Strategies  [foundational RAG]
Chunking splits documents into discrete segments for retrieval, with production pipelines favoring structure-aware and semantic methods over blind fixed-size splits. The core tension is that small chunks yield high retrieval precision but poor reasoning context, while large chunks provide rich context but dilute match accuracy; hierarchical (parent-child) chunking resolves this by indexing small child chunks for search while returning the larger parent to the LLM. Key approaches include recursive splitting at logical boundaries with markdown-aware contextual prepending, semantic chunking via embedding similarity thresholds or cross-encoder segmenters, and content-specific strategies such as AST parsing for code, summarized tables, and VLM-based positional layout embeddings for PDFs. Fixed-size chunking is considered problematic as it breaks semantic units, while modern patterns like Anthropic's Contextual Retrieval prepend global context to each chunk to improve embedding accuracy.
**Key tools/concepts:** Recursive Structure Splitting, Semantic Chunking, Cross-Encoder Segmenters, Hierarchical (Parent-Child) Chunking, Contextual Chunking/Retrieval, Markdown-Aware Splitting, AST Parsing, Summarized Tables, Vision-Language Model (VLM) pre-processing, late chunking

### 03 · Embedding Models  [foundational RAG]
Embedding models convert text into high-dimensional vectors, with the frontier advancing past static single-vector representations to multi-resolution (Matryoshka), late-interaction (ColBERT), and multimodal approaches. Matryoshka Representation Learning (MRL) prioritizes the most important information in the first few dimensions, enabling a 20x reduction in memory and index size with less than a 2% drop in accuracy by allowing initial search on truncated vectors followed by full-resolution refinement. ColBERT v2 shifts from single bi-encoders to token-level late interaction (MaxSim operation), storing one vector per token to achieve significantly higher precision for complex "needle in a haystack" queries. Production indexes leverage in-model binary and Int8 quantization to reduce memory by up to 32x and accelerate search speed using fast Hamming distance calculations, while modern multimodal models allow stacks to treat images, tables, and page layouts as first-class retrieval objects alongside text.
**Key tools/concepts:** Matryoshka Representation Learning (MRL), ColBERT v2 / Late Interaction / MaxSim, Binary and Int8 Quantization, PLAID indexing, Unified vision-text embeddings, ColPali / ColQwen2.5, Vocabulary Mismatch, Bi-Encoders.

### 04 · Vector Databases  [foundational RAG]
Vector databases are purpose-built systems for storing, indexing, and searching high-dimensional embeddings at scale using Approximate Nearest Neighbor (ANN) algorithms. While traditional databases with vector extensions (like PostgreSQL's pgvector) are suitable for up to ~1M vectors, dedicated vector databases are required when scaling to billions of vectors while maintaining sub-100ms P99 latency and complex metadata filtering. The primary architectural tradeoff is between in-memory indexing (like HNSW) which offers the lowest latency but requires massive RAM, and disk-based indexing (like DiskANN) which drastically reduces memory costs at the expense of slightly higher query latency. Selecting the right engine and deployment model (managed serverless vs. self-hosted) depends heavily on an organization's operational capacity, budget, scaling requirements, and multi-tenancy isolation needs.
**Key tools/concepts:** HNSW (Hierarchical Navigable Small World), DiskANN, IVF (Inverted File Index), Product Quantization (PQ), exact vs. approximate search, metadata pre-filtering, distance metrics (Cosine, Euclidean, Dot product), Pinecone, Qdrant, Weaviate, Milvus, Chroma, pgvector, capacity planning.

### 05 · Hybrid Search  [production RAG]
Hybrid search combines dense (semantic) and sparse (keyword) retrieval to leverage the strengths of both approaches, serving as the baseline for production RAG systems. It should be used when queries contain exact matches (like named entities, codes, or version numbers) alongside conceptual phrasing, as dense-only retrieval often misses specific terms while sparse-only retrieval misses semantic similarities. Results from the two pipelines are typically merged using fusion methods like Reciprocal Rank Fusion (RRF) or weighted score fusion, balancing the tradeoff of improved retrieval quality against increased architectural complexity and slightly higher latency. Production implementations can choose between maintaining two separate systems (e.g., a vector DB plus Elasticsearch) or using native single-system hybrid pipelines available in modern databases, while newer techniques like SPLADE offer learned sparse embeddings that unify both retrieval methods in a single storage format.
**Key tools/concepts:** dense and sparse retrieval, BM25/TF-IDF, Reciprocal Rank Fusion (RRF), weighted score fusion, relative score fusion, learned sparse embeddings (SPLADE), alpha tuning (query-adaptive heuristics), staged retrieval, Elasticsearch, Weaviate, Qdrant, Azure AI Search.

### 06 · Reranking Strategies  [production RAG]
Reranking is the second stage of a retrieval pipeline that re-scores a small set of Top 50-100 candidates using a high-precision model (cross-encoder), bridging the gap between efficient first-stage recall and perfect grounding. While bi-encoders embed queries and documents independently for fast O(1) search, cross-encoders jointly process the query-document pair using attention to capture nuanced interactions, at the cost of higher latency (10-100ms). Implementation strategies include multi-stage pipelines (combining sparse, dense, and cross-encoders), LLM-based reranking for complex relevance judgments, and Small Language Model (SLM) distillation to achieve near-LLM quality at low latency. Production systems must balance the cost-benefit tradeoff of latency versus quality, utilizing optimizations like FP16 inference, ONNX/TensorRT, result caching, and fallback strategies.
**Key tools/concepts:** Cross-encoder, Bi-encoder, Two-stage retrieval pipeline, Late interaction, Attention mechanism, BGE-Reranker-v2-m3, Cohere Rerank 3, Jina Reranker v2, Pointwise vs. Listwise LLM reranking, Sliding window reranking, SLM distillation, NDCG, Lost in the middle.

### 07 · GraphRAG  [production RAG]
GraphRAG combines Knowledge Graphs with Retrieval-Augmented Generation to enable multi-hop traversal and global reasoning across datasets where vector similarity fails to connect related entities across documents. Use GraphRAG only when empirical analysis shows 30% or more of retrieval failures are graph-shaped (requiring entity-relationship chains), not for default RAG upgrades. The dominant production pattern in 2026 is graph-as-reranker, which expands vector top-k results via one-to-two-hop graph traversal before cross-encoder reranking, delivering 70-80% of full GraphRAG quality at roughly 20% of upfront cost. Key tradeoffs include expensive LLM-based extraction, a significant quarterly maintenance tail for corpus drift, and scalability limits to millions of entities versus petabytes for vector RAG. Full GraphRAG with community summarization (Microsoft pattern) is reserved for global summarization workloads, while lighter traversal-based variants (HippoRAG, LightRAG) suit local multi-hop queries.
**Key tools/concepts:** Knowledge Graphs, Neo4j, Memgraph, entity-relationship extraction, Leiden community detection, community summarization, graph-as-reranker, multi-hop traversal, hybrid graph-vector search, local vs. global search, SLM-based extraction, LazyGraphRAG, HippoRAG, LightRAG, GraLC-RAG, Microsoft GraphRAG indexing pipeline v2.

### 08 · Agentic RAG  [production RAG]
Agentic RAG replaces the traditional linear retrieve-once pipeline with a dynamic reasoning loop where an agent autonomously decides *when* and *what* to retrieve to resolve complex, multi-step queries. The dominant production patterns include Self-RAG (which uses critic tokens for self-evaluation of relevance and support), Corrective RAG or CRAG (which uses a reliability layer to route to web search or fallbacks if initial retrieval is ambiguous or incorrect), Adaptive RAG (which uses a classifier to select pipeline depth), and multi-hop query decomposition. It should be used when a query requires synthesized proof rather than a simple document match, though engineers must budget for variable latency—typically 8-12s for a 3-4 iteration loop—often mitigated by routing simple queries to faster paths or using speculative retrieval. Production deployments rely on constrained agent frameworks like LangGraph or LlamaIndex Workflows to manage stateful loops and strictly define the graph of possible paths, balancing higher answer quality with the tradeoff of lower reliability and determinism.
**Key tools/concepts:** Reasoning Loop, Self-RAG, Corrective RAG (CRAG), Adaptive RAG, Multi-Hop Reasoning, Agentic Filtering, Sub-Step Plans, Token-Budgeting, Speculative Retrieval, LangGraph, LlamaIndex Workflows, DSPy, Constrained Agent Frameworks.

### 09 · Advanced Retrieval Patterns  [production RAG]
Advanced retrieval patterns address complex query-document gaps in production RAG systems through specialized techniques including query decomposition, hypothetical document generation, contextual chunk enrichment, and in-context reranking. These patterns are used when basic vector similarity search fails due to compound queries, asymmetric query-document length distributions, context dilution in fragmented chunks, or when higher precision is required. Key tradeoffs include increased latency and cost from additional LLM calls (contextual retrieval, HyDE), hallucination risk pulling in semantically similar but incorrect data (HyDE), and the need for large context windows to perform model-based reranking without separate cross-encoder infrastructure.

**Key tools/concepts:** Query Decomposition (Multi-Query), Hypothetical Document Embeddings (HyDE), Contextual Retrieval (Anthropic Pattern), Iterative Document Enrichment, In-Context Reranking, Asymmetric Retrieval, Context Dilution, Reciprocal Rank Fusion (RRF), Hybrid retrieval, Cross-Encoder models, Long Context Reasoning.

### 10 · Contextual Retrieval  [production RAG]

Contextual Retrieval is an ingestion-time technique that uses an LLM to prepend a short explanatory context (50-100 tokens) to each chunk before embedding and indexing, solving the "context dilution" problem where chunks lose meaning when separated from their source document. Pioneered by Anthropic, it reduces top-20 retrieval failures by 35% using Contextual Embeddings alone, 49% when combined with Contextual BM25 (hybrid search), and 67% when paired with a reranker. It is best applied when chunks regularly lose meaning in isolation (e.g., legal contracts, financial reports) and can be skipped if chunks are already self-contained or if real-time, low-latency ingestion is required. The primary tradeoff is ingestion cost and complexity, which can be mitigated by using fast, cheap models, prompt caching (reducing costs by up to 90%), or deterministic Contextual Chunk Headers (CCH) for well-structured documents.

**Key tools/concepts:** Context Dilution, Contextual Embeddings, Contextual BM25, Reciprocal Rank Fusion (RRF), Prompt Caching, Contextual Chunk Headers (CCH), Anthropic, Late Chunking, HyDE.

### 11 · Late Interaction & ColBERT  [production RAG]

Late Interaction (exemplified by ColBERT) is a retrieval paradigm that bridges the gap between fast bi-encoders and highly accurate cross-encoders by encoding queries and documents independently into multi-vector token-level representations that interact via a MaxSim scoring mechanism. It should be used when high precision is required—particularly for domain-specific or term-sensitive searches (like legal or medical)—but the latency of a cross-encoder is unacceptable. The primary tradeoff is infrastructure overhead: while ColBERT achieves cross-encoder-level accuracy at bi-encoder-like speeds, it requires 2-4x the storage of traditional single-vector bi-encoders, practically limiting its use as a primary retriever to corpora under 100M documents or requiring it to be deployed as a second-stage reranker for web-scale systems.

**Key tools/concepts:** ColBERT, ColBERTv2, PLAID (Performance-optimized Late Interaction Driver), MaxSim scoring, residual compression, bi-encoders, cross-encoders, token-level embeddings, RAGatouille, Jina ColBERT v2, PyLate, Vespa, centroid pruning.

### 12 · Multi-Modal RAG  [production RAG]
Multi-modal RAG extends retrieval-augmented generation beyond plain text to process images, tables, charts, and mixed-layout documents, addressing the fact that 40-60% of enterprise content is non-textual. It relies on three dominant architectures: unified vision-text embedding spaces (single index), modality-specific retrieval with fusion (separate indices combined via reciprocal rank fusion), and vision-first page-as-image approaches using late interaction (e.g., ColPali). While unified embeddings offer simplicity, and fusion provides best-in-class precision per modality, the vision-first paradigm eliminates brittle OCR, layout parsing, and table extraction pipelines by treating entire document pages as images. Key trade-offs include high computational and storage costs for multi-vector indices and the challenge of structuring extraction pipelines for complex charts and tables.
**Key tools/concepts:** CLIP, SigLIP / SigLIP 2, ColPali, ColQwen 2.5, ColSmol, Cohere Embed v4, Voyage Multimodal 3, Nomic Embed Vision, Vision-Language Models (Claude, GPT, Gemini), Multi-vector indexing, Late interaction (MaxSim), Table Transformer, Unstructured.io, Tabula, Camelot, Reciprocal Rank Fusion (RRF), Dual-representation strategy.

### 13 · RAG Evaluation Patterns  [production RAG]
RAG evaluation relies on a layered strategy combining the RAG Triad (context relevance, groundedness, answer relevance) for end-to-end correctness, component-level metrics (Recall@K, NDCG, MRR) for debugging, and automated regression testing for CI/CD production safety. The dominant paradigm uses LLM-as-judge to scale evaluations without ground-truth answers, necessitating careful bias mitigation (e.g., preferring binary over scaled scoring, avoiding self-preference) and tiered model usage to manage high operational costs. Robust systems require curated "golden" test sets (seeded with production data and augmented synthetically) and active production monitoring to detect retrieval quality drift over time.
**Key tools/concepts:** RAG Triad (Context Relevance, Groundedness, Answer Relevance), RAGAS (reference-free metrics), LLM-as-Judge, Golden Test Sets, Retrieval Metrics (Recall@K, MRR, NDCG, Precision@K), Automated Regression Testing, Quality Gates, Production Monitoring (Similarity Score Drift, Empty Retrieval Rate), DeepEval, TruLens, UpTrain, Braintrust, LangSmith, DataMorgana.

### 14 · Production RAG at Scale  [production RAG]
Production RAG operates as a distributed system where success depends on optimizing retrieval pipelines, caching layers, routing logic, and multi-tenant isolation under strict latency and cost constraints. Because RAG failures in enterprise deployments occur in retrieval roughly 73% of the time, the knowledge source itself (indexing, chunking, freshness) must be treated as the primary investment rather than the LLM. Winning architectures combine long context windows with RAG (using RAG to find top candidates, then loading them into long context for synthesis), route queries dynamically based on complexity, and enforce strict cost guardrails and tenant isolation (via Silo, Pool, or Bridge models). Key operational challenges include navigating the "lost in the middle" context problem, debugging multi-stage pipeline failures (where 80% of issues trace back to chunking), and scaling vector indexes to millions of documents using sharding and read replicas.
**Key tools/concepts:** Query Routing, Semantic Caching, Parallel Retrieval, Reciprocal Rank Fusion (RRF), Corrective RAG (CRAG), Self-RAG, Adaptive Retrieval, Multi-Index Partitioning, Multi-Tenant Isolation (Silo, Pool, Bridge models), Tiered Model Strategy, End-to-End Trace Logging
